Overall goal
For this BAM, the final desired outputs are:
- read-level outputs from modkit extract calls
- position-level outputs from modkit pileup

but only for:

- CpG-only C-associated modification calls at threshold 0.8

This means we do not want:

That is important because generating all C outputs first and then filtering to CpGs later is much more expensive in disk space and compute time.

What we learned from inspecting the BAM

From:

- C+m?

This means:
- adenine modifications are encoded as A:a
- cytosine modifications are represented by two mod codes:
  - C:m
  - C:h

This BAM therefore does not use the same codes as the earlier histone-mark BAMs that used A:Y and C:Z.

4. Practical interpretation of the mod codes
- A:a corresponds to the adenine modification model output
- C:m and C:h correspond to cytosine modification classes from the 5mCG_5hmCG model

In standard nanopore terminology, these are most naturally interpreted as:
- m = 5mC
- h = 5hmC

Biological decision for the cytosine signal
You now have a real choice:
- keep C:m and C:h separate
- or combine them into one total CpG cytosine-modification signal

Recommended default
Because your stated goal is:
- all A
- CpG-only C

and you did not say that 5mC and 5hmC must remain separated, the most practical default is:
- use A:a for adenine
- combine the C classes with --combine-mods
- restrict the C output to CpG positions with --cpg

This is the cleanest and most efficient workflow if the final desired C output is one CpG methylation-style signal.

Recommended strategy
Use separate modkit runs for A and C:
- one run for all A calls
- one run for combined CpG-only C calls

This is the most efficient workflow for the final biological objective.

Why separate runs are needed
modkit can restrict output to CpGs using:
- --cpg

However, motif filtering applies to the whole command, not only to the C channel.

So if we tried to do both A and C in one CpG-filtered command:
- the C output would be CpG-only
- but the A output would also be restricted by the motif filtering

That is not what we want.

Therefore, the clean approach is:
1. extract calls for A only
2. extract calls for combined C at CpGs only
3. pileup for A only
4. pileup for combined C at CpGs only

Thresholds
Use:
- A threshold = 0.9
- C threshold = 0.8

These thresholds are kept explicit in all commands for consistency.

Step 3. Read-level output for all A calls
Important limitation
Unlike modkit pileup, modkit extract calls does not support:
- --modified-bases

So for read-level output, you cannot ask modkit to emit only A:a directly.

The practical solution is:
- run extract calls once genome-wide with the thresholds
- keep the A rows downstream by filtering on canonical_base = A

This still gives the correct read-level A table, but the A-only selection happens after extraction rather than inside the modkit command.

This produces the genome-wide modkit extract-calls table from which the A-associated rows can be retained downstream.

Command:
modkit extract calls "$bam" "$preproc/extract_calls_all.tsv.gz" \
  --ignore-index \
  --bgzf \
  --filter-threshold A:0.9 \
  --filter-threshold C:0.8 \
  --log-filepath "$preproc/extract_calls_all.log"

What this gives
- read-level calls genome-wide
- with the chosen A and C thresholds explicitly applied

How to obtain the A-only component
Downstream, keep only rows where:
- canonical_base = A

The previous per-read summary script already uses canonical_base, so this is a natural split point.

Step 4. Read-level output for combined CpG-only C calls
For CpG-restricted read-level output, modkit extract calls can restrict to CpGs, but it still does not support --modified-bases.

So this command produces CpG-restricted extract-calls rows, and the downstream C-only selection should again be done using canonical_base = C.

This produces the modkit extract-calls table restricted to CpG sites only.

Command:
modkit extract calls "$bam" "$preproc/extract_calls_cpg.tsv.gz" \
  --ignore-index \
  --bgzf \
  --cpg \
  --reference "$ref" \
  --filter-threshold A:0.9 \
  --filter-threshold C:0.8 \
  --log-filepath "$preproc/extract_calls_cpg.log"

What this gives
- read-level CpG-restricted calls
- no non-CpG output

How to obtain the CpG-only C component
Downstream, keep only rows where:
- canonical_base = C

Then combine or summarize the C:h and C:m rows together at the downstream analysis step if that is your preferred biological interpretation.

If you ever want to keep C:m and C:h separate instead
You do not separate them at the modkit extract-calls command itself.
Instead, after extracting CpG-restricted rows, distinguish:
- call_code = m
- call_code = h

or combine them downstream if you want one total CpG cytosine-modification signal.

Important consequence for downstream read-level summarization
The previous read-level summary workflow used a single combined extract-calls table.

For merged_c1, if the goal is:
- all A
- combined CpG-only C

then the downstream per-read summary should combine:
- the A rows from extract_calls_all.tsv.gz
- the C rows from extract_calls_cpg.tsv.gz

This is preferable to generating a larger combined all-A plus all-C table and discarding non-CpG C later.

Important modkit command distinction

This is an important conceptual point for future debugging and workflow design:
- `modkit extract calls`
  - outputs the predicted modification state for each position in each read
  - for a site with multiple candidate modification classes, there is only one resulting call state per record:
    - modified class 1
    - modified class 2
    - or canonical
- `modkit extract full`
  - produces a table translation of the values stored in the `MM` and `ML` tags
  - keeps the underlying per-modification probabilities
  - so if a read position carries probabilities for both:
    - `5hmC`
    - `5mC`
    both probabilities can be represented in the output

Practical implication for merged_c1
- Because `extract calls` gave repeated trouble for reliable whole-genome merged_c1 extraction, `modkit extract full` became a reasonable fallback candidate to test.
- This is especially relevant because:
  - the BAM is indexed
  - `extract full` is documented to parallelize across aligned genome intervals when an index is present
- If `extract full` is used instead, downstream CNN-preparation code may need slight adaptation because the output schema is not identical to `extract calls`.

```bash

#SBATCH --time=12:00:00
```

5. Practical conclusion
- For `merged_c1`, the operationally successful route was not:
  - one whole-genome `extract full`
- It was:
  - chromosome-level extraction first
  - followed by chunk-level rescue for only the chromosomes that OOMed
- This is the strategy to reuse if a similarly large sample or mark fails on whole-genome extraction.

Generalization to other marks

- The same `extract full` logic can be reused for:
  - `h3k27ac`
  - `h3k4me3`
  - `h3k27me3`
- Practical lesson from `h3k27ac`:
  - even whole-genome chunk arrays can still OOM if chunks are too large for the requested memory
  - a first-pass `50 Mb` chunking strategy on `batch` with `100000M` was still too large for many `h3k27ac` chunks
  - a second-pass retry strategy should therefore split failed `50 Mb` regions into `25 Mb` regions
- This means the reusable operational pattern is:
  1. try whole-genome `extract full` only if the mark is expected to be much smaller than `merged_c1`
  2. if whole-genome fails, switch to chunked extraction
  3. if some chunks still OOM, keep successful chunks and rerun only the failed ones at smaller chunk size

Template command pattern used for all `extract full` jobs

```bash
/data/leuven/383/vsc38330/bin/dist_modkit_v0.6.1_481e3c9/modkit extract full \
  <input.bam> \
  <output.tsv.gz> \
  --bgzf \
  --region <optional_region> \
  --ref /data/leuven/383/vsc38330/thesis_dimelo/src/data/hg38.fa
```

- Important reminders:
  - `--region` is omitted for whole-genome jobs
  - `--region chrN:start-end` is used for chromosome/chunk rescue jobs
  - output files from failed OOM jobs should be removed before reruns
  - successful chromosome/chunk outputs should be kept and reused in the final merge
