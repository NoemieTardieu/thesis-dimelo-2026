Why start with merged_c1 first
- In your current project, merged_c1 gave the cleanest IGV impression and the strongest figure-level behavior.
- That makes it a good first debugging dataset before mixing in the three separate marks.
- Using one sample first reduces moving parts while you confirm:
  - CpG event filtering
  - sequence-window extraction
  - label definition
  - chromosome-based split

Important interpretation
- If you use only merged_c1, then C is constant.
- So this first dataset is best thought of as a debug version of:
  - P(M | D)
- The table still keeps the `cell_type` column so it remains compatible with the later full:
  - P(M | D, C)
- Once this works, you can add more cell types and the same schema still holds.

What this starter workflow does
- Starts from the existing modkit extract-calls file:
  - `/data/leuven/383/vsc38330/thesis_dimelo/modkit_preproc/merged_c1/extract_calls.tsv.gz`
- Keeps only cytosine events:
  - `canonical_base = C`
- Keeps only passing calls:
  - `fail = false`
- Keeps only calls with a valid reference coordinate:
  - `ref_position >= 0`
- Restricts to true CpG cytosines using the hg38 reference:
  - reference dinucleotide at `pos0` must be `CG`
- Extracts a fixed centered DNA sequence window:
  - default = 101 bp
- Defines the label:
  - `m_label = 0` for canonical / unmodified CpG rows
  - `m_label = 1` for `5mC` rows only
  - `5hmC` rows are excluded from the first CNN dataset
- Assigns a chromosome-based split:
  - but should be balanced by per-chromosome event counts rather than by a simple sequential chromosome rule

Why the label is defined this way
- For the first CNN, the target has now been narrowed to:
  - `5mC` only
- This means:
  - canonical / unmodified CpG rows are the negative class
  - `5mC` rows are the positive class
  - `5hmC` rows are not used in the first model
- This decision does not require rerunning `modkit extract full`, because the read-level table still keeps:
  - modification identity via `mod_code`
- So the `m` vs `h` distinction can be enforced later at dataset-building time.

Important split recommendation
- Do not randomize rows across all chromosomes.
- That would risk train / validation / test leakage because nearby loci and highly similar local sequence contexts could land in more than one split.
- A chromosome-based split is still the safer design.
- However, the initial example:
  - train = chr1-16
  - valid = chr17-18
  - test = chr19-22 and chrX
  is probably not the most representative split because the largest chromosomes are concentrated in train.
- A better approach is:
  - estimate the number of final CpG events per chromosome first
  - then assign whole chromosomes to train / valid / test so the total number of events is more balanced
  - keep the assignment fixed in a manifest so the split remains reproducible

Why this is better
- It preserves chromosome-level separation, which reduces leakage.
- It avoids having train dominated by the largest chromosomes while validation and test are built mostly from smaller chromosomes.
- It makes train / valid / test more comparable in total event count and often more stable for model evaluation.

Practical rule
- Keep whole chromosomes intact within one split.
- Randomize or rebalance only at the chromosome level, not at the row level.

- So this is the expected genomic context according to hg38.
- Example idea:
  if the locus in the reference is ACGTG, then ref_kmer is that reference-centered local sequence.

What query_kmer means
- `query_kmer` is the local sequence as represented in the read / query.
- So this reflects what is actually present in the aligned ONT read around that same position.
- It may match the reference exactly, or it may differ because of sequencing errors, mismatches, indels, or alignment effects.

Why both ref_kmer and query_kmer are useful
- `ref_kmer` tells the model the genomic motif / local DNA context.
- `query_kmer` tells the model what the read looked like at that same local region.
- Together they may help the model learn whether some local sequence contexts or read-level patterns are associated with high-confidence vs low-confidence m calls.

  - check whether exact duplicate rows are very common

Integer encoding step completed

To reduce storage and avoid saving gigantic one-hot tensors on disk, the final uppercase files were converted into compact integer-encoded chunked NPZ datasets.

Encoding used:

- A -> 0
- C -> 1
- G -> 2
- T -> 3
- N/other -> 4
- ref_strand: + -> 1, - -> 0
- ref_mod_strand: + -> 1, - -> 0
- pos_norm = forward_read_position / read_length

Encoded outputs now exist here:

- Training NPZ chunks:
  - /staging/leuven/stg_00118/BAM_Noemie/modkit_preproc/merged_c1/tensors_integer_valid5mer_uppercase
- Validation NPZ chunks:
  - /staging/leuven/stg_00118/BAM_Noemie/modkit_preproc/merged_e5b/tensors_integer_valid5mer_uppercase

Why integer encoding was the right choice:

- much smaller on disk than one-hot tensors
- easier to load in chunks
- allows later conversion to embeddings or one-hot in memory during training
- scales better to hundreds of millions of rows


Important observation before CNN training

The encoded chunk manifests show that data are still ordered by class inside the combined files:

- the first chunks are dominated by label 1
- later chunks will contain label 0

This is not an error, but it means the CNN training pipeline must shuffle examples during loading.

So the next step is not another large preprocessing pass. The next step is to build the CNN loader / training pipeline on top of the integer-encoded NPZ chunks.
