==========================

Purpose of this file
--------------------
This file records the important discussion points, environment fixes, local code changes, and next steps related to using HyenaDNA for the thesis project:

Read-level H3K4me3 DiMeLo-seq prediction from DNA sequence.


Current thesis modeling direction
---------------------------------
The current model goal is:

  P(Reg, M | D, C)

where:

  D   = DNA sequence
  M   = endogenous CpG 5mC probability
  Reg = H3K4me3-associated adenine modification / 6mA probability
  C   = optional cell type or sample context

Important correction from promoter discussion:

  In the first implementation, methylation is NOT an input feature.
  Methylation values are training targets/outputs.

Correct first-version input:

  - DNA sequence tokens only: A/C/G/T/N
  - optional cell_type_id or sample_id

Correct first-version outputs:

  - per-position CpG 5mC probability
  - per-position adenine-modification / 6mA probability

This means the first DiMeLo model should not use 5mC or 6mA channels as input
to the HyenaDNA backbone.


Planned architecture
--------------------
Backbone:

  - HyenaDNA-small-32k is the intended main backbone.
  - Suggested checkpoint:
      LongSafari/hyenadna-small-32k-seqlen
      or LongSafari/hyenadna-small-32k-seqlen-hf
  - The 32k model is preferred because it matches typical ONT read/window
    lengths better than the much larger 160k/450k/1M models.

Model structure:

  DNA tokens
    -> HyenaDNA backbone
    -> per-position hidden states [B, L, D]
    -> optional cell/sample embedding
    -> two token-level prediction heads

Hidden-state notation:

  B = batch size, number of reads/windows processed together
  L = sequence length, e.g. 1024 for tiny HyenaDNA or 32768 for 32k HyenaDNA
  D = hidden dimension, the learned feature size per nucleotide position

Example from tiny HyenaDNA smoke test:

  hidden_shape = (1, 1024, 128)

This means:

  1 sequence
  1024 nucleotide positions
  128 hidden features per position

Heads:

  Head 1: CpG 5mC prediction
    - output shape [B, L, 1]
    - loss computed only at valid CpG cytosine positions

  Head 2: H3K4me3-associated 6mA / A-mod prediction
    - output shape [B, L, 1]
    - loss computed only at valid adenine positions

Suggested target tensors per 32k window:

  input_ids       [L]
  target_5mC      [L], float in [0,1], invalid positions ignored
  mask_5mC        [L], 1 at valid CpG targets, else 0
  target_6mA      [L], float in [0,1], invalid positions ignored
  mask_6mA        [L], 1 at valid adenine targets, else 0
  cell_type_id    optional scalar

Loss:

  loss_5mC = masked_loss(pred_5mC, target_5mC, mask_5mC)
  loss_6mA = masked_loss(pred_6mA, target_6mA, mask_6mA)

  total_loss = lambda_5mC * loss_5mC + lambda_6mA * loss_6mA

Start simple:

  - BCEWithLogitsLoss with soft labels in [0,1], or MSE/Huber.
  - Do not binarize methylation for the main training objective.
  - Thresholds can be used later for evaluation and confusion matrices.

Preprocessing direction
-----------------------
The agreed preprocessing starting point is:

  modkit extract full

Why:

  - It preserves read-level modification information.
  - It gives per-read modification probabilities.
  - It avoids pileup/BigWig aggregation, which would collapse single-molecule information.

Important:

  modkit extract full is not directly model-ready.

Needed tensorization step:

  modkit extract full output
    -> group by read/window
    -> reconstruct dense 32k arrays:
         DNA tokens
         target_5mC
         mask_5mC
         target_6mA
         mask_6mA
         metadata

Why BAM is still used with modkit extract full
----------------------------------------------
modkit extract full is the correct read-level modification table, but it is not used alone because the model input is the full DNA sequence.

The BAM provides:

  - the complete read sequence to tokenize for HyenaDNA
  - read length
  - chromosome and alignment coordinates
  - mapping quality
  - strand/orientation information
  - primary vs secondary/supplementary alignment filtering

The extract-full table provides:

  - read_id
  - forward_read_position
  - canonical base
  - mod code
  - continuous modkit probability
  - inferred/non-inferred status

The tensorizer combines both:

  BAM sequence + modkit probabilities -> dense input_ids, targets, and masks

Next thesis-scale roadmap after debug validation
------------------------------------------------
The engineering prototype has now shown that the path works:

  BAM + modkit extract full
    -> read-level tensors
    -> PyTorch Dataset
    -> HyenaDNA tiny
    -> two methylation heads
    -> masked losses and evaluation
    -> HyenaDNA-small-32k forward pass

The next stage is to move from a code/debug dataset to a biologically useful training dataset.

Main next concept:

  Build a better chr16 training dataset from intentionally selected genomic regions rather than the first eligible chr16 reads.

Selection goals:

  - good read coverage
  - real H3K4me3-associated A/6mA heterogeneity
  - CpG methylation variation
  - avoid the chr16 centromere/no-signal gap
  - keep reads <= 32 kb for the first 32k model version
  - later split/crop reads longer than 32 kb

Why this matters:

  The first1000 tensor file is an engineering proof-of-concept. It tests data formats, model inputs, heads, masks, losses, and checkpointing. It is not yet a serious biological training set because it samples the first eligible reads in genomic order.

Immediate implementation plan:

  1. Rank chr16 100 kb bins using existing merged_e5b A-mod and CpG bigWig
     tracks, plus BAM read-count support.
  2. Exclude the central chr16 centromere/no-signal interval.
  3. Write a BED file of selected high-signal/high-heterogeneity regions.
  4. Re-run tensorization using only reads overlapping those selected regions.
  5. Use the resulting .npz as the first better 32k training/debug dataset.

Backbone-freezing explanation
-----------------------------
In the current 32 kb training runs, the HyenaDNA backbone is frozen.

Meaning:

  - The pretrained HyenaDNA sequence model weights do not change.
  - HyenaDNA acts as a fixed sequence feature extractor.
  - Only the two small output heads are trained:
      hidden states -> 5mC prediction
      hidden states -> 6mA prediction

Why this was chosen first:

  - It is much cheaper and faster.
  - It reduces overfitting risk on the small chr16 prototype dataset.
  - It validates whether pretrained sequence representations already contain
    useful information for the DiMeLo targets.
  - It is a safer first step before fine-tuning the full 32 kb backbone.

What it does not test yet:

  - Whether HyenaDNA internal sequence filters should adapt to DiMeLo data.
  - Whether partial or full fine-tuning improves 5mC/6mA prediction.

Later model stages can compare:

  1. frozen backbone + trained heads
  2. partially unfrozen top layers + trained heads
  3. full fine-tuning, if enough data and compute are available

  The sequence-only HyenaDNA-small-32k baseline works as an engineering
  baseline. It can ingest read-level DiMeLo tensors, train two methylation heads,
  and produce non-random held-out region metrics.

  However, it is not yet a final biological-performance model. The model is
  currently sequence-only and uses a frozen HyenaDNA backbone. It captures some
  average signal and some per-position ranking, but held-out regions show
  systematic shifts, especially for CpG 5mC. For example, several test regions
  have target_minus_pred around 0.10-0.15, meaning the model underpredicts their
  mean methylation level.

    DNA sequence alone.

  - The current model does not use methylation or chromatin context as input.
  - The HyenaDNA backbone is frozen, so only the output heads adapt to DiMeLo
    data.
  - The dataset is still limited to selected chr16 regions from merged_e5b.

  - Add sample_id/cell_type_id once multiple samples or cell contexts are used.
  - Compare frozen backbone with partially unfrozen or fully fine-tuned HyenaDNA.
  - Add stronger baselines and ablations, such as local CNN and shorter-context
    HyenaDNA.
  - Expand from chr16 to more chromosomes and both merged_c1 and merged_e5b.
  - Consider later multimodal inputs, such as using observed methylation context
    for masked reconstruction tasks, while keeping the first supervised version
    sequence-only.
  - Evaluate per-region and per-feature behavior, not only global metrics.

  easier and more sequence-driven task. 6mA/H3K4me3 remains harder and probably needs
  more data, multiple samples/cell contexts, or future multimodal/chromatin-aware
  modeling to improve substantially.

Sample-conditioned combined model setup
---------------------------------------
Date: 2026-05-18

Goal:

  Upgrade the combined E5B+C1 pooled model so it uses sample_id instead of ignoring
  it. This is the first implementation step toward the thesis notation:

    P(Reg, M | D, C)

  where C is represented here by sample_id.

Created Python scripts:

  train_region_split_hyenadna_two_head_sample_conditioned.py
  evaluate_region_split_hyenadna_two_head_sample_conditioned_extended.py

Implementation:

  - Dataset reads sample_id from the combined NPZ files.
  - Model creates a learned sample embedding:
      merged_e5b -> sample_id 0
      merged_c1  -> sample_id 1
  - HyenaDNA produces per-position hidden states from DNA sequence.
  - The sample embedding is expanded across positions and concatenated to the
    per-position Hyena hidden state before the two prediction heads.
  - Output heads still predict:
      CpG 5mC probability per position
      adenine 6mA probability per position

  run_train_combined_top100_region_split_small32k_a100_sample_conditioned_lastblock.sbatch
  run_eval_combined_top100_region_split_small32k_a100_val_sample_conditioned_lastblock_extended.sbatch

  - HyenaDNA-small-32k-seqlen
  - max length 32768
  - batch size 1
  - 3 epochs
  - up to 300 train batches per epoch
  - up to 150 val batches per epoch
  - sample embedding dim 16
  - num samples 2
  - head/sample-embedding lr 1e-3
  - backbone lr 1e-5
  - unfreeze backbone.layers.3 and backbone.ln_f

Promoter discussion: next modeling directions after 4-chromosome HyenaDNA prototype
- Discussion date: 2026-05-21.
- Main point: instead of only adding more chromosomes, improve the biological design of the dataset and the conditional modeling formulation.

1. Handle reads longer than 32 kb
- Current pipeline skips reads longer than 32,768 bp.
- Proposed improvement: split long reads into multiple <=32 kb windows instead of discarding them.
- Two possible strategies:
  - Non-overlapping chunks: take bases 0-32768, then 32768-65536, etc., until the read is exhausted.
  - Overlapping windows: use sliding windows, e.g. 32 kb window with 8 kb or 16 kb stride, to avoid losing signal at chunk boundaries.
- Biological/technical reason: long nanopore reads may contain useful long-range methylation/chromatin information; skipping them wastes data and may bias training toward shorter reads.

2. Centromere / low-signal regions as negative-control regions
- Current region selection avoids centromere/no-signal regions.
- Promoter suggestion: test whether the model predicts low/no signal in regions where there should be little signal.
- Key question: are centromeric regions truly empty, or do they have read coverage but little/no modification signal?
- If there is read coverage but low 5mC/6mA signal, they can be useful negative controls.
- If there are no reads, they are not useful for model testing because there is no data to evaluate.

3. Cross-chromosome and cross-sample generalization
- Instead of only adding more chromosomes to train and test together, test generalization explicitly.
- Proposed experiments:
  - Train on one chromosome and validate/test on another chromosome.
  - Train on chromosomes from one BAM/sample and validate/test on another BAM/sample.
  - Train on one sample/chromosome combination and test on another sample/chromosome combination.
- Biological/technical reason: this tests whether the model learned general sequence/context rules or just chromosome/sample-specific patterns.

4. Return to conditional model P(Reg | D, C, M)
- Current model is closer to P(Reg, M | D, C): it predicts 5mC and 6mA/H3K4me3 jointly from DNA sequence and sample context.
- New proposed model: use CpG 5mC methylation as an input feature, together with DNA sequence and context, and predict only regulatory/chromatin signal Reg, i.e. 6mA/H3K4me3.
- This would use one output head: 6mA probabilities with 6mA mask.
- Biological reason: DNA methylation may help explain chromatin/regulatory state, so conditioning on observed or predicted M could improve Reg prediction.
- Comparison to make:
  - Current model: P(Reg, M | D, C)
  - New model: P(Reg | D, C, M)
  - Evaluate whether including 5mC as input improves 6mA/H3K4me3 prediction.

5. Related-output decoder / shared-parameter multi-task architecture
- Current decoder uses two mostly independent linear heads for 5mC and 6mA after a shared HyenaDNA representation.
- Promoter idea: use a decoder architecture with more shared parameters because 5mC and 6mA/H3K4me3 are biologically related.
- Possible direction:
  - Shared decoder trunk after HyenaDNA, then task-specific heads.
  - Cross-task conditioning: predict Reg from D,C,M and also flip direction to predict M from D,C,Reg.
  - Compare P(Reg | D,C,M) and P(M | D,C,Reg) to study relationship between DNA methylation and chromatin/regulatory signal.
- Biological reason: methylation and histone/regulatory marks are not independent; explicitly modeling their relationship may improve prediction and interpretation.

Suggested priority from this discussion
1. Add long-read chunking/windowing to stop discarding >32 kb reads.
2. Build negative-control/centromere coverage check to distinguish no reads from reads-with-no-signal.
3. Run cross-chromosome generalization experiments.
4. Implement P(Reg | D,C,M) by adding 5mC as input and predicting only 6mA/H3K4me3.
5. Later improve decoder architecture with shared task-specific parameters and bidirectional conditional tests.

Implementation note: overlapping windows for reads >32 kb
- Added explicit long-read handling to make_chr16_dimelo_tensors.py.
- Previous behavior remains default: --long-read-mode skip.
- New modes:
  - --long-read-mode truncate: keep only the first 32 kb window.
  - --long-read-mode nonoverlap: split long reads into consecutive non-overlapping <=32 kb chunks.
  - --long-read-mode overlap: split long reads into sliding windows, e.g. 32 kb window with 16 kb stride.
- Added metadata columns: window_start, window_end, window_length.
- For overlapping windows, modkit forward_read_position is remapped to the local window position. A modification can appear in more than one overlapping window if it falls in the overlap region.
- This preserves previous datasets and allows clearly labeled new datasets such as overlap16k.

Literature-aligned decoder roadmap
- The stronger modelling idea is not just to add a shared MLP, but to move toward a small multi-task sequence-to-epigenome decoder, inspired by Basenji/Enformer-style multi-track sequence prediction and chromatin-mark cooperation models.
- Main objective: keep the final target as P(Reg,M | D,C), but conceptualize it as a small multi-omics panel. Current outputs are 5mC and 6mA/H3K4me3-like signal. If additional reliable tracks become available later, add them as extra heads rather than building separate models.
- Architecture roadmap:
  1. Keep HyenaDNA as the long-context DNA encoder.
  2. Keep sample_id/cell/condition embedding and broadcast it over sequence positions.
  3. Replace the independent linear heads with a shared sequence-aware decoder block, not only a pointwise MLP. Candidate block: LayerNorm -> lightweight local mixing layer (depthwise convolution, gated convolution, or small Hyena-like block) -> GELU MLP -> residual connection. Use only 1 layer first.
  4. Use task heads after the shared decoder: 5mC head and 6mA head. Heads can remain linear initially.
- Explicit coupling ideas:
  1. Latent epigenetic factor bottleneck: decoder output -> small K-dimensional latent factors -> task heads. This forces the model to express tasks through shared epigenetic axes before splitting into 5mC and 6mA.
  2. Latent 5mC-to-6mA FiLM conditioning: compute a 5mC-specific latent representation from decoder features, then use it to generate gamma/beta modulation for the 6mA representation. This lets 6mA use the latent 5mC context without requiring observed 5mC labels at inference time.
- Training roadmap:
  1. Joint multi-task loss L = w_5mC L_5mC + w_6mA L_6mA.
  2. Because 6mA positives are rare, test task weighting or focal loss for 6mA. The goal is to improve 6mA AUPRC/AUROC without destroying 5mC.
  3. Optional staged training: first stabilize trunk/decoder on abundant 5mC, then jointly finetune with 6mA. This should be an ablation, not the first implementation.
- Evaluation roadmap:
  1. Compare against current best linear two-head P(Reg,M | D,C).
  2. Compare against shared MLP smoke models already run.
  3. Evaluate chr16 overlap test first, then fair overlap chr19/chr11/chr17.
  4. Stratify later by genomic annotations (promoters/enhancers/gene bodies/repeats/CpG islands) if annotation tracks are available.
- Practical implementation order:
  1. Finish the current MLP128/lr3e-4 smoke, because it is already prepared.
  2. If it does not clearly improve, do not keep trying pointwise MLP sizes.
  3. Implement the first sequence-aware shared decoder: one residual Conv/GLU decoder block plus two heads, with optional 6mA task weighting.
  4. Only after that, test latent bottleneck and FiLM coupling.
- Interpretation: this plan is better than only using a shared MLP because it aligns with the core lesson of multi-task regulatory sequence models: multiple tracks should shape a shared sequence representation. It also keeps the final model usable at inference, because 5mC remains an output rather than a required input.
